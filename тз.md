[FEATURE] подготовить backend-инфраструктуру для перехода с кастомной UUID-аутентификации на полноценный Spring Security с входом по email и password.

Описание
Сейчас backend использует легаси-механизм: клиент передаёт X-Auth-Token, а сервер интерпретирует его как UUID пользователя. Нужно заменить эту основу на стандартную security-модель
Spring, чтобы дальнейшая авторизация во всех модулях строилась на штатных механизмах framework, а не на кастомных interceptor-ах и ручной обработке заголовков.

В рамках задачи нужно:

подключить Spring Security в backend-зависимости;
добавить SecurityFilterChain;
определить публичные и защищённые endpoint'ы;
добавить PasswordEncoder для безопасной проверки паролей;
реализовать загрузку пользователя и ролей в формате, совместимом со Spring Security;
реализовать backend-аутентификацию по email + password;
подготовить principal/authentication-модель, которую потом смогут использовать остальные backend-модули;
исключить UUID из X-Auth-Token как способ входа в новой схеме;
заложить базу для дальнейшего перевода модульной авторизации на Spring Security.
Что не входит в задачу:

полный перевод всех модулей проекта на новую схему авторизации;
переработка frontend-части;
UI/flow для смены или сброса пароля;
полная миграция всех существующих тестов на новую security-модель;
окончательное удаление всего legacy-кода, если он ещё нужен на переходный период.


Какую проблему решает?
Текущая схема аутентификации небезопасна и архитектурно хрупкая: UUID пользователя фактически используется как готовый токен доступа. Это создаёт риск компрометации доступа, не позволяет
опираться на стандартные механизмы Spring Security и усложняет масштабирование backend-логики безопасности.

Без перехода на нормальную backend-аутентификацию проект остаётся завязан на кастомную реализацию, которую придётся вручную поддерживать и расширять во всех модулях. Это повышает
стоимость любых изменений в правах доступа, ухудшает предсказуемость поведения security-слоя и затрудняет дальнейшее развитие системы.


Критерии приёмки
- В backend подключён Spring Security и приложение стартует с новой security-конфигурацией.
- В проекте добавлен SecurityFilterChain с явным разделением публичных и защищённых endpoint'ов.
- Для backend-аутентификации используется email + password, а не UUID из заголовка.
- Для хранения и проверки паролей используется PasswordEncoder.
- Пользователь и его роли загружаются в формате, совместимом со Spring Security.
- После успешной аутентификации backend может определить текущего пользователя в новой security-схеме.
- UUID-токен больше не является способом входа в новой backend-инфраструктуре.
- Реализация пригодна как базовый слой для дальнейшего перевода существующей модульной авторизации на Spring Security.
