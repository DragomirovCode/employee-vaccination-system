Title: [FEATURE] Реализовать RBAC для модулей auth/employee/vaccination/reporting

Описание
Реализовать полноценную ролевую авторизацию (RBAC) на backend в соответствии с документацией:
- Роли и зоны ответственности из `docs/business-analytics/functionality/README.md`:
  - `PERSON`: доступ только к своим данным.
  - `HR`: просмотр сотрудников своего подразделения и отчетов, без редактирования медицинских данных.
  - `MEDICAL`: управление вакцинациями/документами и справочником вакцин.
  - `ADMIN`: полный доступ и управление ролями.
- Подключить проверку прав на уровне HTTP-эндпоинтов и/или сервисного слоя.
- Убрать демонстрационную проверку `dev-token` и перейти на реальную модель пользователя/ролей из модуля `auth`.
- Ограничить доступ к отчетам (`reporting`) согласно роли и департаменту.
- Добавить инфраструктуру получения текущего пользователя (по токену/заголовку в dev-режиме) и его ролей.
- Логику доступа сделать расширяемой: новая роль/право добавляется без переписывания бизнес-сервисов.
- Обновить `backend/README.md` и описать матрицу прав.

Изменения в UI
- [x] UI не затрагивается
- [ ] Приложен скриншот "Как сейчас"
- [ ] Приложен макет/скриншот с пометками "Как должно стать"

Какую проблему решает?
Сейчас авторизация фактически отсутствует (демо-проверка токена), поэтому нельзя обеспечить разграничение доступа к персональным и медицинским данным. Это создает риск утечки данных, нарушения внутренних политик доступа и несоответствия требованиям аудита и безопасности.

Альтернативные способы решения
- Временный gateway-фильтр без ролей в приложении: быстро, но не покрывает предметные ограничения (свой сотрудник/свой департамент).
- Проверки только в UI: не подходит, так как не защищает backend API.
- Хардкод ролей в контроллерах без централизации: усложняет поддержку и расширение.

Критерии приёмки
1) В `auth` реализована загрузка пользователя и его ролей из `users` + `user_roles` + `roles` для каждого запроса.
2) Демонстрационная проверка `dev-token` удалена из основного контура доступа.
3) Для эндпоинтов `vaccination`/`documents` доступ на запись есть только у `MEDICAL` и `ADMIN`.
4) Для эндпоинтов отчетности (`/reports/*`) доступ есть у `HR`, `MEDICAL`, `ADMIN`; `PERSON` не имеет доступа.
5) Для данных сотрудников действует ограничение видимости:
   - `PERSON` видит только собственные данные.
   - `HR` видит сотрудников только своего подразделения (включая дочерние, если включена иерархия).
   - `ADMIN` видит все.
6) Операции управления ролями доступны только `ADMIN`.
7) На неавторизованный запрос API возвращает `401`, на недостаток прав — `403`.
8) Добавлены и проходят интеграционные тесты матрицы ролей (минимум: `PERSON`, `HR`, `MEDICAL`, `ADMIN`) по ключевым эндпоинтам.
9) В `backend/README.md` добавлена секция с матрицей прав и примером передачи учетного контекста в запросе.
10) Текущий функционал модулей (`employee`, `vaccine`, `vaccination`, `reporting`, `audit`) не ломается: все существующие тесты проходят.
